# Contract v30：权威相邻地形交互查询

状态：协议 1.30 / contract-v30 active baseline

## 已完成边界

- 协议新增 `TerrainInteractionDto`，包含动作种类、方向、目标位置、稳定 terrain ID、是否需要检定、当前是否可用和可选不可用原因。
- 当前动作种类为 `open-door`、`close-door`、`bash-door`；不可用原因先覆盖 `occupied-by-actor` 与 `occupied-by-item`。
- `GameSnapshot` 与每次 `GameUpdate` 都输出当前位置的 `terrainInteractions`。旧协议数据缺失该字段时按空列表读取。
- 核心固定按北、东北、东、东南、南、西南、西、西北扫描；同一格按开门、关门、破门排序。只输出该 terrain 声明支持的动作，不为普通墙和破损门伪造动作。
- 锁门在查询中同时提供开门和破门，并标记 `requiresCheck: true`；普通关闭门的开门不需要检定，普通开启门只提供关门。
- 占用判断与实际关门/破门规则共用同一核心入口。怪物优先于物品成为不可用原因，避免同一格多种阻挡时结果随容器遍历变化。
- 查询是派生视图，不进入 save 或 state hash。terrain、角色、怪物和地面物品仍是唯一权威状态。
- 前端进入 O/C/B 模式前先确认附近存在相应动作；选择方向后再次匹配权威条目。方向不适用或被占用时只显示本地化原因，不发送必然失败的命令，也不消耗回合。

## 与原版 RFB v1.3.0.7 对比

原版 `src/cmd2.c` 的 easy-open 路径会通过 `count_dt()` / `count_chests()` 扫描相邻格；只有一个关闭门或箱子时自动填入方向。普通 `do_cmd_open()`、`do_cmd_close()` 与 `do_cmd_bash()` 则在取得方向后读取相邻 feature flags 和占用状态，决定是否执行、攻击怪物或输出文本。

相同点：

- 都以玩家当前位置和八个相邻格为交互候选范围；
- 都由地形 feature/terrain 能力决定可以开、关或撞，而不是由前端硬编码地图字符；
- 都在执行前考虑怪物和物品占用；
- 都能在只有明确门目标时减少无意义的方向尝试。

主动差异：

- 重构输出稳定、强类型、可测试的交互列表，由前端选择是否使用；原版 easy-open 直接修改全局 `command_dir`，没有独立查询协议。
- 重构查询不执行行动、不消费 RNG、不改变回合，也不自动替玩家选择目标；原版 easy-open 会在唯一候选时自动定向并继续原命令流程。
- 重构同时声明动作是否需要检定，但不暴露检定骰和内部能力值；原版在命令执行函数内直接读取 feature power 与玩家全局技能。
- 重构不可用原因目前只覆盖角色/物品占用；原版还会把怪物阻挡转换为近战，并有箱子、卡死门、混乱能量、旅行和重复命令分支。
- 重构列表按跨平台固定顺序进入协议和 exact fixtures；原版依赖邻格扫描函数和全局命令状态，没有 DTO、回放检查点或状态哈希边界。

本切片复刻的是“先从相邻 feature 能力确定可交互目标，再执行方向命令”的关系，但把自动命令选择改造成无副作用的权威查询。
