# Contract v34：多深度楼层连接

状态：协议 1.34 / contract-v34 active baseline

## 已完成边界

- 世界内容由单个 `proceduralFloor` 扩展为按深度排序的 `proceduralFloors`。每层声明稳定 ID、`returnFloorId`，并可用 `nextFloorId` 与 `downStairTerrainId` 连接下一层。
- 内容编译器验证楼层 ID 唯一、引用存在、下一层反向返回当前层且深度严格加一；有下一层时必须同时声明下楼梯 terrain。
- 深度 1 在入口西侧生成稳定下楼梯，深度 2 首次进入时独立消费 RNG 并生成自己的 terrain、怪物和物品。
- 离开楼层时继续保存准确玩家位置。深度 2 上楼会恢复深度 1 的下楼梯位置，再移动到深度 1 上楼梯可恢复地表入口位置。
- save v1 的当前层和 `storedFloors` 已能保存任意已声明楼层，因此不增加存档字段；state-hash Schema v15 已覆盖所有离层状态。

协议 1.34，内容包 1.28.0，content hash `8319b75e64585ef782358ed5287e087d14fab3626dfa854296696751f66896ac`，active baseline 72 个 exact fixtures。

## 与原版 RFB 对比

相同点：原版以地牢深度和上下楼方向连接连续楼层；首次进入新深度生成楼层，返回保存过的楼层时恢复其状态；上下楼都属于明确的楼层切换行动。

主动差异：重构不用隐式整数深度推断目的地，而使用稳定 `FloorId` 和显式有向连接，并把每层状态放入可哈希、可回放的 `FloorState`。这允许后续加入分支、任务层和非线性地牢，而不改变核心存档身份。

暂未实现：分支连接和独立到达点、任务/临时层、旧层淘汰、随机楼梯、回忆与传送入口、守护者及最终层规则。旧 v33 已生成的深度 1 不会补下楼梯。
