# 授权、版权与素材迁移审计

状态：P0 边界与新仓库双许可证已确定

> 本文是工程迁移边界，不是法律意见。授权不明确的文件默认不可复制到新仓库。

## 1. 当前结论

新仓库目前包含原创代码、规划文档、最小内容包和原创演示 SVG tileset，没有复制旧 C 源码、旧 `localization/` 内容、发布压缩包、字体、音频或旧 tileset。

旧项目不是一个可以整体视为 MIT、Apache 或单一 GPL 的代码库。初步检查发现：

- 多数古老 Angband 源文件包含“仅教育、研究和非营利用途”的 Angband 授权声明；
- 部分文件提供 GPL v2 或 Angband 授权二选一；
- `jsmn.h` 等第三方文件使用 MIT；
- 部分 Win32 文件包含 Microsoft 版权声明；
- 数据、帮助文本、专名、tileset、字体和其他资产没有统一的机器可读授权清单；
- GitHub 无法为旧仓库识别出统一许可证。

已确定的仓库边界：

- 不复制旧源码实现；
- 不直接搬运旧版图片、字体、音效和大段文本；
- 不把旧数据文件整体作为新仓库初始内容提交；
- 不在新游戏发行包中携带旧 RFB 内容；
- 开发工具通过 `RFB_LEGACY_SOURCE=D:/codex/Frogcomposband/master` 只读访问本地旧仓库；
- 固定读取 `v1.3.0.7`，不读取带未提交改动的当前工作树；
- 可以把旧程序当作本地黑盒行为参考，并编写原创协议、测试和实现。

## 2. 迁移分类

### A：可直接进入新仓库

- 本仓库原创 Rust、TypeScript、测试和设计文档；
- 本仓库原创的旧存档读取器和行为对照工具；
- 新制作且作者明确授权的图像、音频和字体；
- 许可证兼容且保留声明的第三方依赖；
- 不包含旧表达形式的行为断言、接口和数据 Schema。

### B：仅限本地参考，不进入新仓库

- 旧游戏规则数据、数值表和地图生成参数；
- 怪物、物品、神器、法术、任务的名称与描述；
- 帮助文档和剧情文本；
- 旧存档样本、解析结果和转换后的内容包；
- 旧 tileset、图标、字体和声音；
- 从旧源码改写而来的算法。

这些内容只用于本地兼容分析和行为对照。工具输出写入 `.local/`，不得进入新仓库或新游戏发行包。

### C：默认禁止复制

- 授权未知的二进制文件；
- 无来源说明的网络图片、字体和音效；
- Microsoft 示例代码或其他限制性代码的直接复制；
- 旧发布包中的 DLL 和工具链文件；
- 不能确认再分发权利的第三方 tileset。

## 3. Clean-room 边界

新核心默认采用“行为对照、独立实现”原则：

1. 先从旧版可观察行为生成测试场景；
2. 新代码按协议和行为要求独立实现；
3. 不逐行翻译 C 函数，不保留旧变量布局和控制流；
4. 如果必须参考具体算法，先记录来源和授权结论；
5. 代码评审中标记任何从旧源码移植或改写的部分。

这项原则不自动解决游戏名称、内容数据、剧情和美术的授权问题，这些资产仍需单独审计。

## 4. 已建立的许可证结构

```text
LICENSES/
├─ README.md
└─ CC-BY-SA-4.0.txt

LICENSE
NOTICE
assets/provenance.json
packs/README.md
```

来源记录建议格式：

```json
{
  "id": "asset.tileset.example",
  "path": "assets/tiles/example.png",
  "source": "https://example.invalid/source",
  "authors": ["Author Name"],
  "license": "CC-BY-4.0",
  "licenseFile": "LICENSES/third-party/CC-BY-4.0.txt",
  "modified": true,
  "notes": "裁切并重新打包图集"
}
```

## 5. 新依赖准入规则

- Cargo、npm 和 Tauri 依赖必须保留 lockfile；
- CI 使用 `cargo-deny` 和 npm 许可证审计生成报告；
- 默认允许 MIT、BSD-2-Clause、BSD-3-Clause、Apache-2.0、ISC、Unicode-3.0 等常见兼容许可证；
- GPL/LGPL/MPL、非商业、禁止修改和自定义许可证必须人工审核；
- 运行时依赖、构建依赖和仅开发依赖分别记录；
- 不允许仅因包管理器可以下载，就假设资源可以随安装包重新分发。

## 6. 正式发布前门槛

- 保持旧内容不进入仓库和发行包；
- 所有打包资产都有来源和授权；
- 桌面安装包包含必要的第三方 notices；
- GitHub 仓库根目录提供 `LICENSE` 或明确的多许可证说明；
- CI 对新增依赖和资产来源执行检查。

## 7. 剩余未决决策

1. 首个正式字体和音效的来源；
2. 项目名称与标识是否采用单独商标政策；
3. 哪些兼容测试结果可以在不包含旧表达的前提下公开；
4. 是否需要联系上游维护者补充历史授权说明。

这些决策不阻止原创架构、协议、测试工具和最小演示开发，但任何正式发行包都必须继续排除旧版和授权不明资产。

当前原创 `web/public/tilesets/image-demo/tiles.svg` 已登记到 `assets/provenance.json`，按 CC BY-SA 4.0 授权；它只用于验证 manifest、atlas 切片和资源回退，不包含旧版图像或第三方素材。

## 8. 新仓库双许可证

- 原创 Rust/TypeScript 代码：`MPL-2.0`；
- 原创内容数据、文档和美术：`CC BY-SA 4.0`；
- 第三方内容：保留原许可证和来源文件；
- 旧版内容：不进入本仓库和新游戏发行包；
- 项目名称与标识是否作为商标管理，另行决定。

根目录 `LICENSE` 保存 MPL-2.0 完整文本，`LICENSES/CC-BY-SA-4.0.txt` 保存 CC BY-SA 4.0 完整文本，具体目录适用范围见 `LICENSES/README.md`。任何文件都可以通过 SPDX 标识或相邻声明指定不同许可证。
