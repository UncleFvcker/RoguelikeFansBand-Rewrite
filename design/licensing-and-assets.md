# 授权、版权与素材迁移审计

状态：P0 初步审计完成，最终许可证待确认

> 本文是工程迁移边界，不是法律意见。授权不明确的文件默认不可复制到新仓库。

## 1. 当前结论

新仓库目前只包含原创规划文档，没有复制旧 C 源码、旧 `localization/` 内容、发布压缩包、字体、音频或 tileset。

旧项目不是一个可以整体视为 MIT、Apache 或单一 GPL 的代码库。初步检查发现：

- 多数古老 Angband 源文件包含“仅教育、研究和非营利用途”的 Angband 授权声明；
- 部分文件提供 GPL v2 或 Angband 授权二选一；
- `jsmn.h` 等第三方文件使用 MIT；
- 部分 Win32 文件包含 Microsoft 版权声明；
- 数据、帮助文本、专名、tileset、字体和其他资产没有统一的机器可读授权清单；
- GitHub 无法为旧仓库识别出统一许可证。

因此，在完成逐类审计前：

- 不给新仓库擅自声明一个覆盖旧内容的宽松许可证；
- 不复制旧源码实现；
- 不直接搬运旧版图片、字体、音效和大段文本；
- 不把旧数据文件整体作为新仓库初始内容提交；
- 可以把旧程序当作黑盒行为参考，并编写原创协议、测试和实现。

## 2. 迁移分类

### A：可直接进入新仓库

- 本仓库原创 Rust、TypeScript、测试和设计文档；
- 新制作且作者明确授权的图像、音频和字体；
- 许可证兼容且保留声明的第三方依赖；
- 不包含旧表达形式的行为断言、接口和数据 Schema。

### B：审计后迁移

- 旧游戏规则数据、数值表和地图生成参数；
- 怪物、物品、神器、法术、任务的名称与描述；
- 帮助文档和剧情文本；
- 旧存档导入代码；
- 旧 tileset、图标、字体和声音；
- 从旧源码改写而来的算法。

每项必须记录来源文件、原作者、许可证、修改方式、是否包含可版权保护的表达，以及新仓库中的目标位置。

### C：默认禁止复制

- 授权未知的二进制文件；
- 无来源说明的网络图片、字体和音效；
- Microsoft 示例代码或其他限制性代码的直接复制；
- 旧发布包中的 DLL 和工具链文件；
- 不能确认再分发权利的第三方 tileset。

## 3. Clean-room 边界

新核心默认采用“行为对照、独立实现”原则：

1. 先从旧版可观察行为生成测试场景；
2. 新代码按协议和行为要求独立实现；
3. 不逐行翻译 C 函数，不保留旧变量布局和控制流；
4. 如果必须参考具体算法，先记录来源和授权结论；
5. 代码评审中标记任何从旧源码移植或改写的部分。

这项原则不自动解决游戏名称、内容数据、剧情和美术的授权问题，这些资产仍需单独审计。

## 4. 计划建立的授权清单

```text
LICENSES/
├─ project-license.txt
├─ third-party/
└─ legacy/

NOTICE
assets/provenance.json
content/provenance.json
```

来源记录建议格式：

```json
{
  "id": "asset.tileset.example",
  "path": "assets/tiles/example.png",
  "source": "https://example.invalid/source",
  "authors": ["Author Name"],
  "license": "CC-BY-4.0",
  "licenseFile": "LICENSES/third-party/CC-BY-4.0.txt",
  "modified": true,
  "notes": "裁切并重新打包图集"
}
```

## 5. 新依赖准入规则

- Cargo、npm 和 Tauri 依赖必须保留 lockfile；
- CI 使用 `cargo-deny` 和 npm 许可证审计生成报告；
- 默认允许 MIT、BSD-2-Clause、BSD-3-Clause、Apache-2.0、ISC、Unicode-3.0 等常见兼容许可证；
- GPL/LGPL/MPL、非商业、禁止修改和自定义许可证必须人工审核；
- 运行时依赖、构建依赖和仅开发依赖分别记录；
- 不允许仅因包管理器可以下载，就假设资源可以随安装包重新分发。

## 6. 正式发布前门槛

- 确定新仓库原创代码的项目许可证；
- 对旧内容建立可追溯清单；
- 所有打包资产都有来源和授权；
- 桌面安装包包含必要的第三方 notices；
- GitHub 仓库根目录提供 `LICENSE` 或明确的多许可证说明；
- CI 对新增依赖和资产来源执行检查。

## 7. 未决决策

1. 新原创代码采用何种许可证；
2. RoguelikeFansBand 名称和旧版内容数据是否按衍生作品方式继续发布；
3. 哪些旧规则必须通过移植实现，哪些可以 clean-room 重写；
4. 首个官方 tileset、字体和音效的来源；
5. 是否需要联系上游维护者补充授权说明。

在这些决策完成前，本仓库可以继续开发原创架构、协议、测试工具和最小演示，但不能发布包含授权不明旧资产的正式游戏包。

## 8. 新仓库许可证候选

建议把代码与内容资产分开授权：

| 选择 | 特点 | 适合情况 |
|---|---|---|
| `MPL-2.0` | 修改过的源码文件必须继续开放，但可与其他许可证模块组合 | 当前首选，兼顾开放性、插件生态和桌面发行 |
| `GPL-3.0-or-later` | 分发的组合衍生程序通常需要整体提供对应源码 | 希望最大程度阻止闭源分叉 |
| `Apache-2.0` | 宽松、包含明确专利授权，允许闭源再利用 | 最重视采用率和商业兼容 |
| `MIT` | 极简宽松，但专利和贡献条款较少 | 小型库或不在意闭源分叉 |

当前建议方案：

- 原创 Rust/TypeScript 代码：`MPL-2.0`；
- 原创内容数据、文档和美术：`CC BY-SA 4.0`；
- 第三方与旧版迁移内容：保留原许可证和来源文件；
- 项目名称与标识是否作为商标管理，另行决定。

在用户明确确认前不创建最终 `LICENSE`。如果未来直接合入 GPL 或旧 Angband 授权代码，必须重新评估组合发布方式；选择 MPL 并不会改变被导入内容原有的限制。
