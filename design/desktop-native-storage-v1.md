# 桌面原生存档与诊断 v1

状态：Windows/Tauri 垂直切片已实现

## 1. 范围

本阶段在保留手动 `.rfbsave` 导入/导出的同时，为桌面版增加应用私有存档槽。TypeScript 只管理槽名称和用户交互；目录选择、事务写入、备份恢复、删除和诊断日志由 Tauri Rust 适配层负责。游戏核心和 `.rfbsave` 容器仍不感知操作系统路径。

当前接口：

- `list_native_saves`：列出槽摘要及 `ready`、`recoverable`、`corrupt` 状态；
- `save_native_game`：创建新槽或安全覆盖指定槽；
- `load_native_game`：校验并载入正式文件或最近有效备份；
- `delete_native_save`：删除正式文件及三份备份。

## 2. 目录与身份

- 存档根目录由 Tauri `app_local_data_dir()` 提供，并在其下使用 `saves/`；
- 日志目录由 Tauri `app_log_dir()` 提供，文件名为 `rfb-desktop.log`；
- 前端不接收也不显示绝对路径；
- 槽 ID 由原生层生成，格式为 `save-<Unix 毫秒>`，碰撞时增加数字后缀；
- 槽 ID 只允许 ASCII 小写字母、数字和连字符，最长 80 字节；
- 玩家可见名称写入 `SaveHeaderV1.slot_name`，去除首尾空白后必须包含 1 至 80 个非控制字符。

`slot_name` 是向后兼容的 Header 元数据字段，旧存档缺失时按空字符串读取；它不进入权威 payload 或 state hash。

## 3. 写入事务与备份

所有原生存档命令通过独立互斥锁串行执行。一次写入按以下顺序完成：

1. 在存档目录创建唯一临时文件；
2. 写入完整 `.rfbsave` 字节并调用 `sync_all`；
3. 重新读取临时文件，验证容器校验和并构建临时游戏快照；
4. 将 `.bak2` 轮换为 `.bak3`、`.bak1` 轮换为 `.bak2`、正式文件轮换为 `.bak1`；
5. 将临时文件 rename 为正式文件；
6. 再次读取并验证已提交文件；
7. 启动时删除上次异常退出遗留的隐藏 `.tmp` 文件。

如果最终 rename 失败且正式文件已被轮换，原生层会尝试把 `.bak1` 恢复为正式文件。任何阶段失败都返回结构化错误，不会先删除旧正式存档。

## 4. 列表、载入与恢复

列表扫描正式文件和 `.bak1` 至 `.bak3`，但只向前端返回逻辑槽，不暴露文件名以外的目录信息：

- `ready`：正式文件有效；
- `recoverable`：正式文件无效或缺失，但至少一份备份有效；
- `corrupt`：正式文件和备份都无法验证。

载入按正式文件、`.bak1`、`.bak2`、`.bak3` 顺序尝试。使用备份时 UI 会明确显示恢复来源；载入后的快照同时同步 TypeScript 传输层的 revision 和 command sequence，保证下一条命令继续使用正确序列。

删除操作需要前端确认，并移除正式文件及所有备份。损坏槽仍可覆盖或删除，但不能直接载入。

## 5. 错误、日志与隐私

Tauri Commands 返回 `{ code, detail }`：

- `code` 是稳定、可本地化的分类；
- `detail` 用于开发诊断，不直接展示给玩家；
- 前端把名称错误、缺失、损坏和目录不可用映射为 Fluent 文本；
- 存档错误不会把正常运行的核心连接状态改为“核心错误”。

`rfb-desktop.log` 使用 Unix 毫秒时间戳记录启动、存档成功、备份恢复、删除及错误代码。panic hook 只记录脱敏后的源文件名、行号和列号，不写存档内容、绝对路径、玩家文本或网络凭据。日志只保存在本机，不自动上传。

活动会话标记和自动 `.rfbdiagnostic` 报告见[桌面崩溃诊断闭环 v1](crash-diagnostics-v1.md)。玩家不需要主动导出日志：Rust panic 和未正常退出在下次启动自动生成报告，前端未处理异常立即生成报告；正常退出清理标记。

## 6. 兼容入口

顶部的手动“导出存档/载入存档”继续保留，用于备份、迁移和跨机器传输。原生槽和手动文件使用同一 `.rfbsave` 编解码器；本阶段没有引入第二种存档格式，也没有复制旧 RFB 存档内容。

## 7. 验证

Rust 单元测试覆盖：

- 多次覆盖后的备份轮换；
- 正式文件损坏后的 `.bak1` 恢复；
- 正式文件及备份删除；
- 路径穿越式槽 ID 和无效名称拒绝。

Windows Tauri E2E 覆盖：

- 新建、列出、元数据和状态显示；
- 保存后移动并恢复原回合、位置和 state hash；
- 恢复后继续派发命令，验证传输序列同步；
- 覆盖和删除；
- 原有手动导入/导出链路继续工作；
- 所有操作保持同一个地图 Canvas 节点。
- 合成前端未处理异常自动生成 `.rfbdiagnostic`，并显示不自动上传的 Fluent 提示。

## 8. 当前限制与后续

- 当前 UI 不提供“另存为”或槽重命名；
- 备份数量固定为 3，尚未成为用户设置；
- 不提供手动日志导出按钮；自动报告 v1 尚不包含 Windows minidump、完整存档或最近命令回放；
- Linux/macOS 依赖相同 Tauri 路径接口，但尚未建立桌面 E2E；
- Android 暂不接入这套桌面槽 UI，未来应使用应用私有目录和系统文件选择器适配。
